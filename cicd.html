üöß CI/CD and GitHub Actions

Introduction to GitHub Actions

GitHub Actions is a powerful CI/CD platform that allows you to automate your software development workflows directly in your GitHub repository. It enables you to build, test, and deploy your code right from GitHub.

Key Benefits

  • Automate repetitive tasks
  • Ensure code quality through automated testing
  • Deploy applications automatically
  • Integrate with thousands of third-party services
  • Respond to GitHub events (push, pull requests, issues, etc.)

Core Concepts

1. Workflow

A workflow is an automated process that you define in your repository. Workflows are made up of one or more jobs and can be triggered by events, scheduled, or manually.

2. Jobs

A job is a set of steps that execute on the same runner. By default, jobs run in parallel, but you can configure them to run sequentially with dependencies.

Jobs Execution Pattern

Example with job dependencies:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      # steps here
  
  test:
    needs: build  # This job waits for build to complete
    runs-on: ubuntu-latest
    steps:
      # test steps here
  
  deploy:
    needs: [build, test]  # Waits for both jobs
    runs-on: ubuntu-latest
    steps:
      # deployment steps here

3. Steps

Steps are individual tasks that run commands or actions within a job. Steps execute in order and are sequential within a job.

Example steps:

steps:
  - name: Checkout code
    uses: actions/checkout@v3
  
  - name: Set up Node.js
    uses: actions/setup-node@v3

      node-version: '18'

Runners

A runner is a server that executes your workflows. Runners listen for available jobs, run one job at a time, and report progress, logs, and results back to GitHub.

GitHub Events Deep Dive

GitHub Actions workflows are triggered by events that occur in your repository. Understanding event activity types and filters is crucial for creating efficient, targeted workflows.

Common Events and Activity Types

1. pull_request Event

on:
  pull_request:
    types: [opened, synchronize, reopened, closed]

Available activity types:

  • opened - PR is created
  • synchronize - New commits pushed to the PR branch
  • reopened - Previously closed PR is reopened
  • closed - PR is closed (merged or not)
  • ready_for_review - Draft PR marked as ready

2. Event Filters

Branch and Tag Filters
on:
  push:
    branches:
      - main
      - develop
      - 'releases/**'  # Pattern matching
    paths:
      - 'src/**'
      - 'package.json'
    paths-ignore:
      - '**.md'
      - 'docs/**'

üí° Filter Best Practices

  • Use specific paths to avoid unnecessary runs
  • Select only needed activity types
  • Combine filters for precision
  • Use paths-ignore for documentation changes

      ~/.npm

    restore-keys: |

  • ‚úÖ Use secrets for all sensitive data
  • ‚úÖ Rotate secrets regularly
  • ‚úÖ Use environment-specific secrets
  • ‚úÖ Set minimal required permissions
  • ‚ùå Never print or echo secret values

Controlling Workflow and Job Execution

Workflow Control Overview

flowchart TD A[Workflow Controls] --> B[Concurrency] A --> C[Conditions] A --> D[Manual Controls] A --> E[Timeouts] B --> B1[Cancel in-progress runs] B --> B2[Queue workflows] C --> C1[if conditionals] C --> C2[Status functions] D --> D1[workflow_dispatch] D --> D2[Approval gates] E --> E1[Job timeouts] E --> E2[Step timeouts] style A fill:#e1f5ff,stroke:#0366d6,stroke-width:3px style B fill:#fff5e6,stroke:#fd7e14,stroke-width:2px style C fill:#e6ffe6,stroke:#28a745,stroke-width:2px style D fill:#f0e6ff,stroke:#6f42c1,stroke-width:2px

Concurrency Control

# Cancel previous runs when new commits are pushed
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

Conditional Execution

steps:
  # Only run if tests succeed
  - name: Deploy
    if: success()
    run: echo "Tests passed, deploying..."
  
  # Only run if tests fail
  - name: Notify failure
    if: failure()
    run: echo "Tests failed!"
  
  # Always run (even if previous steps fail)
  - name: Cleanup
    if: always()
    run: rm -rf temp/

‚≠ê Status Functions

  • success() - All previous steps succeeded
  • failure() - Any previous step failed
  • always() - Always runs (regardless of status)
  • cancelled() - Workflow was cancelled

      - name: Check Node version

      - name: Run tests
        run: npm test

Composite Action Example

name: 'Setup Node.js Application'
description: 'Checkout code, setup Node.js, and install dependencies'

inputs:
  node-version:
    description: 'Node.js version to use'
    required: true
    default: '18'

outputs:
  cache-hit:
    description: 'Whether the cache was hit'
    value: ${{ steps.cache.outputs.cache-hit }}

runs:
  using: 'composite'
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: 'npm'
      id: cache
    - shell: bash
      run: npm ci

Pattern 1: Mono-repo CI/CD

jobs:
  detect-changes:
    outputs:
      frontend: ${{ steps.changes.outputs.frontend }}
      backend: ${{ steps.changes.outputs.backend }}
    steps:
      - uses: dorny/paths-filter@v2
        id: changes
        with:
          filters: |
            frontend:
              - 'packages/frontend/**'
            backend:
              - 'packages/backend/**'
  
  build-frontend:
    needs: detect-changes
    if: needs.detect-changes.outputs.frontend == 'true'
    steps:
      - run: npm run build --workspace=frontend

Pattern 2: Multi-Environment Pipeline

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm ci && npm test && npm run build
  
  deploy-dev:
    needs: build
    environment: development
    steps:
      - run: echo "Deploy to dev"
  
  deploy-staging:
    needs: deploy-dev
    environment: staging
    steps:
      - run: echo "Deploy to staging"
  
  deploy-production:
    needs: deploy-staging
    environment: production  # Requires manual approval
    steps:
      - run: echo "Deploy to production"
